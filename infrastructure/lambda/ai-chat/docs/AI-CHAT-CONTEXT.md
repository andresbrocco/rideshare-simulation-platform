## Project Overview

This platform is a purpose-built data engineering demonstration combining a real-time discrete-event simulation engine with a full medallion lakehouse analytics pipeline. It simulates a rideshare marketplace operating in São Paulo, Brazil — one of the largest urban mobility markets in the Americas — producing a continuous stream of synthetic events that feed two parallel architectural paths: a real-time visualization path for live map rendering, and a batch analytics path for historical analytics and business intelligence.

### Why a Rideshare Simulation?

Rideshare operations are an ideal domain for demonstrating data engineering at scale because they produce a rich, interrelated event vocabulary from a single coherent domain. A single completed trip generates at minimum: a request event, one or more driver offer events, a driver assignment, a sequence of GPS pings (one every few simulated seconds for the duration of the trip), status transitions for both driver and rider, a payment event, and a rating event. These events span multiple Kafka topics, require coordination across multiple processing paths, and feed both real-time visualization and analytical aggregations.

The São Paulo setting adds authentic geographic richness. The simulation city is divided into 96 distinct districts matching the real metropolitan geography, each with demand weights derived from real-world urban mobility patterns — high-demand districts in the financial center and entertainment zones, lower-demand residential periphery, and distinct rush-hour demand concentrations that differ from weekend leisure patterns. Drivers have preferred operating zones encoded in their DNA that are biased toward high-demand areas, creating realistic supply-demand imbalances that drive the surge pricing feedback loop. This geographic specificity means the H3 spatial matching, OSRM road routing, and surge price calculations all operate on geometrically meaningful coordinates rather than random noise.

The project demonstrates competencies across the full data engineering stack without requiring a live production system or real customer data. ## Architecture & Technology Stack

### System Layers

The platform is organized into five distinct layers, each producing or consuming data from the next:

**Layer 1 — Simulation Engine.** A Python SimPy discrete-event simulation generates all synthetic rideshare activity. It runs alongside a FastAPI control plane in the same process, with the SimPy event loop on a background thread and the HTTP server on the main thread. A thread-safe command queue (`ThreadCoordinator`) bridges the two concurrency models. The simulation is the single source of truth for all domain events — nothing else in the system generates or modifies events.

**Layer 2 — Event Streaming.** Apache Kafka (KRaft mode, no ZooKeeper) serves as the central event bus with eight application topics: `trips`, `gps_pings`, `driver_status`, `surge_updates`, `ratings`, `payments`, `driver_profiles`, and `rider_profiles`. A Confluent Schema Registry enforces JSON Schema Draft 2020-12 contracts at publish time. GPS pings receive 8 partitions (twice the 4 used by other topics) due to higher message volume. All events carry distributed tracing correlation fields linking them across session, business entity, and causal chain.

**Layer 3 — Dual-Path Processing.** The same Kafka events feed two independent consumer groups simultaneously. The real-time path (Stream Processor) applies 100-millisecond windowed GPS aggregation, deduplicates via Redis `SET NX`, and publishes to Redis pub/sub channels consumed by the simulation API's WebSocket server. The batch path (Bronze Ingestion) persists raw events verbatim as partitioned Delta Lake tables on S3/MinIO, routing malformed messages to per-topic dead-letter queue tables.

**Layer 4 — Medallion Lakehouse.** Three Bronze, Silver, and Gold Delta Lake layers are orchestrated by Apache Airflow. DBT runs hourly Silver incremental transforms (JSON parsing, deduplication, enum validation) and daily Gold full-refresh transforms (star schema with SCD Type 2 dimensions). Great Expectations validates both layers after each run. Trino provides distributed SQL access to all three layers via the Hive Metastore or AWS Glue catalog.

**Layer 5 — Visualization and Analytics.** Grafana serves six dashboard categories spanning five datasources: Prometheus (infrastructure and simulation metrics), Trino (Gold-layer business intelligence), Loki (log aggregation and pipeline observability), Tempo (distributed traces), and the Airflow metadata PostgreSQL database (pipeline lineage and run history). A React SPA with deck.gl provides a live geospatial map of driver and trip positions driven by WebSocket updates at approximately 10 Hz.

### Technology Choices and Trade-offs

**SimPy over actor frameworks.** SimPy's generator-based coroutines are simpler to reason about than actor-model frameworks (Akka, Ray) because control flow is explicit: each agent process is a Python generator that `yield`s when waiting and resumes when the event fires. This transparency is valuable for a demonstration system where the simulation logic itself should be readable. The trade-off is that SimPy runs single-threaded within its event loop, so parallelism comes from the simulation speed multiplier (advancing simulated time faster than wall-clock time) rather than from hardware parallelism.

**KRaft Kafka over ZooKeeper Kafka.** KRaft mode eliminates the ZooKeeper dependency, reducing the minimum container footprint by one service and simplifying the bootstrap sequence. This matters for a demo platform that starts and stops frequently. The trade-off is that KRaft is a relatively recent addition to Kafka's production support surface, but this is acceptable given the demo context.

**Delta Lake over Apache Iceberg or Hudi.** Delta Lake is chosen for its mature Python bindings via `delta-rs` (Rust-backed, no JVM required for writes), straightforward DuckDB integration (the `delta_scan()` function reads Delta tables natively), and strong compatibility with both local development and AWS Glue. The `delta-rs` library allows Bronze Ingestion to write Delta tables without running a JVM Spark process, which is important for the local development stack's container footprint.

**DuckDB local / Glue production dual-target DBT.** Running DBT against DuckDB locally means developers can test transformations without spinning up Spark infrastructure. The same model files run against Glue Interactive Sessions in production. Cross-database SQL differences are abstracted via `adapter.dispatch` macros, so the business logic in the model files is clean of dialect-specific branches. This is a deliberate ergonomic choice that prioritizes developer experience while maintaining production-grade execution.

**Trino FILE authenticator over LDAP or OAuth.** The Trino password file approach keeps visitor credential management entirely within the existing provisioning Lambda workflow. Visitor bcrypt hashes are written to Secrets Manager during Phase 1 provisioning and rendered into the Trino password file at container startup. This is significantly simpler than operating an LDAP server or OAuth provider for a demo platform, at the cost of requiring a container restart to add new users. Since users are provisioned before the platform comes up (and provisioned users are stable for the session duration), this limitation is acceptable.

**OpenTelemetry Collector as multi-signal hub.** Rather than having each service push directly to Prometheus, Loki, and Tempo, all application services push metrics and traces to a single OTel Collector via OTLP/gRPC. The collector routes metrics to Prometheus via remote write, reads Docker log files for Loki shipping, and forwards traces to Tempo. This centralizes signal routing logic in a single configuration file, simplifies service code (each service just emits OTLP), and enables future signal routing changes without touching application code.

**Tempo-generated span metrics and service graph.** Tempo processes incoming traces and derives two additional Prometheus metric sets: `traces_spanmetrics_*` (RED metrics — rate, error, duration — per service-operation pair) and `traces_service_graph_*` (request rates and durations between service pairs). These are remote-written back to Prometheus, giving Grafana the ability to show trace-derived metrics in the same dashboards as infrastructure metrics, and to navigate from a slow Prometheus metric directly to representative traces in Tempo using exemplars.

## Simulation Engine

### Discrete-Event Model

The simulation uses SimPy's generator-based discrete-event framework. Each agent (driver or rider) runs as a SimPy process — a Python generator that `yield`s `simpy.Timeout` events to advance simulated time. The `env.run()` loop processes these events in chronological order, advancing the simulation clock to the next scheduled event on each step. Because the loop is single-threaded, there are no race conditions between agent processes — each step is atomic. The simulation can run faster than real-time by setting a speed multiplier that controls how many simulated seconds pass per wall-clock second, allowing significant data accumulation for analytics without waiting for wall-clock time. The Performance Controller adjusts this multiplier dynamically based on infrastructure backpressure signals.

Each driver process is a generator function that repeatedly cycles through: check for commands in the spawn queue → select behavior based on current state → execute the behavior (which may involve yielding for a computed duration) → emit Kafka events → loop. This coroutine style makes the state machine transitions explicit in the code structure: a driver "waiting for a rider offer" is literally waiting at a `yield env.timeout(offer_wait_seconds)` line, and the acceptance or rejection is computed when the generator resumes.

### DNA-Based Agent Behavior

Each agent's behavior is governed by a frozen, immutable DNA model — a Pydantic model with `frozen=True` — generated at spawn time using Faker with Brazil locale for realistic synthetic attributes. The DNA model is the complete specification of how an agent will behave for the duration of its existence in the simulation.

**Driver DNA parameters include:**
- `acceptance_rate` (float 0–1): base probability of accepting an offered trip; modified by trip characteristics at decision time
- `preferred_zones` (list of district IDs): weighted preference for operating zones during repositioning
- `experience_level` (enum): novice/intermediate/expert; affects rating probability distributions and acceptance behavior
- `vehicle_type` (enum): economy/comfort/premium; determines fare multiplier and rider matching eligibility
- `active_hours` (tuple of hour ranges): time windows during which the driver prefers to be online
- `min_trip_rating` (float): threshold below which the driver declines to continue accepting trips (DNA-level attrition modeling)
- `home_zone` (district ID): attraction zone for repositioning after trips

**Rider DNA parameters include:**
- `request_frequency` (float, mean inter-request interval in simulated minutes): governs how often the rider generates trip requests
- `destination_preferences` (weighted district list): probability distribution over trip destinations
- `patience_seconds` (float): maximum wait time before cancellation; riders with low patience cancel fast when drivers are scarce, driving up surge
- `tip_probability` (float): base probability of adding a tip to the payment event
- `vehicle_preference` (enum or None): optional preference for vehicle type; None means any type is acceptable
- `rating_bias` (float): positive or negative shift applied to the base rating distribution

DNA is immutable during a simulation run. Mutations to agent attributes — such as driver rating changes after a trip — do not modify the DNA model. Instead, they emit new `driver_profiles` Kafka events that feed SCD Type 2 dimension updates in the Gold layer, preserving the full history of attribute changes for analytics. This ensures the simulation is fully reconstructible from its event log.

Because DNA is frozen, behavioral decisions are pure functions of the agent's DNA plus environmental state (current time, nearby demand, offer details). This makes agent behavior deterministic given the same DNA and environment, which is valuable for testing and reproducibility. The test suite uses seeded Faker instances to generate deterministic DNA for all unit tests.

### H3 Geospatial Matching

Spatial matching between riders requesting trips and nearby available drivers uses H3 hexagonal spatial indexing at resolution 9 (approximately 174-meter edge length, or roughly a city block). This resolution was chosen to balance lookup precision against the number of cells required to cover São Paulo's metropolitan area. At resolution 7 (the coarser option used for aggregation in some contexts), cells are approximately 1.2 km across — too coarse for pickup matching. At resolution 11, cells are 24 meters across — too fine, requiring large k-rings and excessive candidates. Resolution 9 gives a lookup footprint of about 5-10 city blocks, which corresponds to realistic driver proximity expectations.

Available drivers are indexed by their current H3 cell in a dictionary keyed by `h3_index`. This enables O(1) retrieval of all drivers currently located in any given cell, making the matching lookup sub-millisecond even with thousands of active drivers. When a rider requests a trip, the matching server:

1. Converts the rider's latitude/longitude to an H3 cell index at resolution 9
2. Queries drivers in that cell and the surrounding k-ring (initially k=5, approximately 800 meters of coverage)
3. If no candidates are found, doubles the k-ring radius (k=10, then k=20, etc.) until candidates appear or the search radius exceeds a configured maximum
4. Applies a bounding-box pre-filter to discard candidates whose Haversine distance exceeds a threshold before computing the full composite score
5. Scores remaining candidates using a weighted composite of: estimated time of arrival (primary weight), driver rating (secondary), historical acceptance rate from DNA (tertiary)
6. Offers the trip to the highest-scoring driver; if rejected or timed out, tries the next candidate and increments `offer_sequence`

The `offer_sequence` counter in the trip event records how many drivers were attempted. This is a valuable analytics signal: high `offer_sequence` values indicate supply scarcity in a zone even if the trip eventually completes, driving surge pricing and zone demand heat calculations.

The Haversine formula computes great-circle distance from two coordinate pairs using the spherical Earth approximation. For GPS coordinates within São Paulo's 55 km × 40 km extent, the approximation error is under 0.1%, which is negligible for matching purposes. The bounding-box pre-filter (checking coordinate deltas before computing the trigonometric Haversine) eliminates the most distant candidates cheaply.

### OSRM Road-Network Routing

Route geometry for driver navigation comes from OSRM (Open Source Routing Machine), a high-performance C++ routing engine loaded with São Paulo's OpenStreetMap road network data. The OSRM instance runs as a Docker container with the São Paulo PBF file pre-processed during image build, exposing a JSON API for route requests.

OSRM routes are queried for two legs of each trip: the driver-to-pickup route (when the driver is assigned) and the pickup-to-destination route (the passenger trip). Each route response returns a polyline geometry, total duration, and total distance. The simulation GPS engine then interpolates driver positions along the polyline at each simulated time step, producing realistic curvilinear GPS traces that follow actual São Paulo roads rather than straight lines.

Routes are cached by H3 cell-pair keys in an LRU cache with 10,000 entries and TTL expiry enforced on read. The cache key is `(origin_h3_cell_resolution_9, destination_h3_cell_resolution_9)`, snapping coordinates to cell centers before the OSRM request. This approach trades slight geometric imprecision (routes are from cell center to cell center, not exact coordinates) for a high cache hit rate at high simulation speed multipliers, when many drivers may traverse the same major corridors concurrently. An LRU cache of 10,000 entries covers roughly the cartesian product of São Paulo's major O-D pairs without excessive memory consumption.

### Driver State Machine

Drivers cycle through a fully specified state machine enforced by `VALID_TRANSITIONS` dictionaries:

- **OFFLINE**: Driver is not available. Transitions to AVAILABLE when the driver comes online.
- **AVAILABLE**: Driver is idle and seeking riders. The matching server may offer a trip. Transitions to EN_ROUTE_PICKUP on acceptance.
- **EN_ROUTE_PICKUP**: Driver is navigating to the rider's pickup location. GPS pings are emitted along the OSRM route. Transitions to AT_PICKUP on arrival.
- **AT_PICKUP**: Driver has arrived at the pickup location and is waiting for the rider to board. Transitions to ON_TRIP when the rider boards, or back to AVAILABLE if the rider cancels during this window.
- **ON_TRIP**: Driver is carrying the passenger to the destination. GPS pings continue along the destination route. Transitions to REPOSITIONING on arrival.
- **REPOSITIONING**: Driver has completed the trip and is driving toward their preferred home zone without a passenger. Transitions back to AVAILABLE when repositioning is complete.

Each state transition emits a `driver_status` Kafka event. The `REPOSITIONING` state is significant for analytics: it contributes to `online_minutes` (the driver is logged in) but not to `on_trip_minutes`, and it represents supply moving toward zones rather than toward demand. Drivers with home zones in low-demand areas can create persistent supply deficits in high-demand zones if repositioning attractors are not calibrated.

### Rider State Machine

Riders have a simpler lifecycle:

- **IDLE**: Rider is between trips, waiting for the inter-request interval (drawn from an exponential distribution parameterized by `request_frequency` DNA).
- **REQUESTING**: Rider has submitted a trip request and is waiting for a driver offer.
- **WAITING_FOR_DRIVER**: Driver has been assigned; rider is waiting at the pickup location.
- **ON_TRIP**: Rider is in the vehicle en route to destination.
- **CANCELLED**: Rider cancelled during REQUESTING or WAITING_FOR_DRIVER (transitions to IDLE after a cooldown).
- **COMPLETED**: Trip completed; rider submits rating and payment events and transitions back to IDLE.

Cancellation during REQUESTING occurs if the patience timeout expires before a driver accepts. Cancellation during WAITING_FOR_DRIVER occurs if the driver takes longer than the rider's patience allows. Both cancellation paths emit a `trips` Kafka event with `status=CANCELLED` and include the cancellation reason (TIMEOUT_OFFER vs TIMEOUT_PICKUP) for analytics.

### Surge Pricing Feedback Loop

A surge pricing calculator runs as a periodic SimPy process that monitors zone-level supply-demand ratios. For each H3 zone at resolution 7 (coarser resolution used for surge aggregation — approximately 1.2 km hexagons covering several resolution-9 cells), it computes:

```
demand = count of active REQUESTING riders in zone
supply = count of AVAILABLE drivers in zone
surge_multiplier = base_fare_multiplier * (1 + surge_factor * max(0, demand/supply - 1))
```

When `demand/supply` exceeds 1.0, the surge multiplier rises above 1.0. The `surge_factor` DNA parameter (configured at the system level, not per-agent) controls the responsiveness of surge to demand-supply imbalance. Surge updates are published to the `surge_updates` Kafka topic and streamed to the frontend, which renders zone-level demand heat via deck.gl hexagon aggregation layers. The visualization uses a color ramp from blue (low demand/surge) through yellow to red (high surge), giving a visual representation of real-time market state.

Higher fares from surge pricing feed back into driver behavior: the acceptance scoring function gives a bonus to offers in high-surge zones, which tends to attract more drivers toward high-demand areas over time, gradually reducing the imbalance. This feedback loop is a core demonstration of supply-demand dynamics in two-sided marketplaces.

### Traffic and GPS Modeling

The traffic model applies time-of-day multipliers to OSRM base durations, simulating São Paulo's notorious traffic patterns:

- Morning rush (7:00–9:00 AM): +40% duration multiplier
- Evening rush (17:00–19:00): +40% duration multiplier
- Late night (22:30–5:30 AM): -15% duration multiplier (reduced congestion)
- All other hours: 1.0× (baseline)

These multipliers are applied as scale factors to OSRM's precomputed route durations, which already reflect São Paulo road network characteristics. The result is that drivers take noticeably longer during rush hours and travel faster late at night, producing realistic variation in trip completion times and GPS ping densities.

The GPS simulator interpolates driver positions along route polylines using precomputed cumulative arc lengths. At each simulated time step, it computes the current position along the polyline using linear interpolation between waypoints, then applies Gaussian noise with a 10-meter standard deviation (clipped at 15 meters maximum displacement) to simulate GPS receiver error. Occasional signal dropout frames (position unchanged for 1-2 steps) simulate real-world GPS behavior. The result is GPS traces that follow São Paulo roads with realistic noise, rather than the mathematically perfect interpolation that would be implausible in real telemetry.

### Simulation Lifecycle Management

The simulation engine has a four-state lifecycle enforced by the same `VALID_TRANSITIONS` pattern used by agents:

- **STOPPED**: No SimPy process running. The background thread has exited.
- **RUNNING**: SimPy event loop active. Agents are generating events.
- **DRAINING**: Pause requested but not yet complete. The SimPy event loop continues stepping to allow in-flight trips to reach terminal states (COMPLETED or CANCELLED) naturally. New trip requests are suppressed during DRAINING, but existing trips continue to completion.
- **PAUSED**: SimPy loop is halted. State is preserved. Can be resumed by transitioning back to RUNNING.

The two-phase pause is deliberately designed to avoid truncated trip records in the analytics pipeline. A naive pause (stopping the SimPy thread immediately) would leave many trips in intermediate states — EN_ROUTE_PICKUP, AT_PICKUP, ON_TRIP — which would appear in Bronze as events without corresponding COMPLETED or CANCELLED terminal events. Silver deduplication would not catch this (deduplication is on `event_id`, not trip completeness), and Gold `fact_trips` would contain partial trips with null `dropoff_*` fields and incorrect completion rates.

The DRAINING state resolves this by allowing the simulation to wind down naturally. The draining phase completes when either: (a) all active trips have reached terminal states, or (b) a forced cancellation timeout fires after 7,200 simulated seconds — at which point any remaining in-flight trips are force-cancelled and the engine settles into PAUSED. The control panel shows a draining indicator during this phase.

### ThreadCoordinator and Cross-Thread Safety

The simulation's two-thread model requires careful coordination. The SimPy event loop runs on a background thread. The FastAPI HTTP server runs on the main thread. Cross-thread state mutations flow through the `ThreadCoordinator` command queue — a `queue.Queue` that accepts typed command objects (`PauseCommand`, `ResumeCommand`, `SpawnDriverCommand`, etc.).

The SimPy step loop drains the command queue at each tick, executes queued commands in the SimPy thread's context, and signals completion via a `threading.Event`. FastAPI handlers enqueue commands and optionally wait for the completion event before returning the HTTP response. This ensures that HTTP-level commands always execute in the correct thread context.

`env.process()` — SimPy's mechanism for registering new agent coroutines — is not thread-safe and must be called from the SimPy thread. HTTP requests to spawn new drivers or riders enqueue `SpawnDriverCommand` / `SpawnRiderCommand` objects, which the SimPy thread drains via four dedicated spawner processes (one per agent type, running at `simpy.INFINITY` priority). This decouples HTTP agent registration from SimPy process creation, allowing burst spawning via HTTP without blocking the event loop or violating thread safety.

Cross-thread state reads use frozen dataclasses (`AgentSnapshot`, `TripSnapshot`, `SimulationSnapshot`). The SimPy thread writes new snapshot instances atomically by replacing the reference; the FastAPI thread reads the current reference without acquiring any lock. Because Python's GIL makes reference assignment atomic for CPython, this lock-free read pattern is safe in practice — though the design explicitly acknowledges the CPython dependency.

## Data Flow & Medallion Lakehouse

### Kafka Event Bus Design

The simulation publishes to eight Kafka topics, each scoped to a single event type. This one-topic-per-event-type design was chosen over a single firehose topic to enable per-topic consumer group management, per-topic schema evolution, and selective Bronze ingestion parallelism (GPS topics can have more partitions and faster consumer throughput without affecting the other topics).

All topics operate at-least-once delivery semantics. The producer uses `acks=1` (leader acknowledgment), meaning messages are durably recorded but not yet committed to all replicas at the time of producer acknowledgment. This is a deliberate trade-off: `acks=all` would add latency that slows the SimPy event loop at high speed multipliers, while `acks=0` would risk message loss on leader failure. For a demo simulation, `acks=1` is the appropriate balance.

Events are JSON Schema-validated via the Confluent Schema Registry at publish time. Validation failures are logged as structured warnings rather than halting the simulation. This non-fatal validation design prevents a schema mismatch — which might occur if the simulation code is updated while the schema registry still has an old schema version — from crashing the event loop. The simulation logs a structured warning with the event type, validation error message, and payload hash, making schema drift detectable without service interruption.

A configurable malformed event rate can inject additional corrupted copies of events alongside clean ones. This is used in integration testing and demo sessions to exercise the Bronze ingestion DLQ pipeline end-to-end, verifying that malformed events are routed to DLQ tables rather than being silently discarded.

### Schema Registry and Event Contracts

All cross-service data contracts live in a centralized schema directory. JSON Schema Draft 2020-12 definitions for all eight Kafka event types are registered with the Schema Registry and validated on both publish (by the simulation producer) and consume (optionally, by downstream consumers). The schema for each event type includes:

- **Correlation fields**: `event_id` (UUID), `session_id`, `correlation_id` (business entity ID, typically `trip_id`), `causation_id` (ID of the triggering event)
- **Timestamp fields**: `event_timestamp` (simulated time, ISO 8601), `published_at` (wall-clock producer time)
- **Domain payload**: event-specific fields with strict type and enum constraints

The `EventFactory` chains causation by reading the previous event's ID from the trip record and writing the new event's ID back, forming an ordered causal chain for the entire trip lifecycle. This enables downstream systems to reconstruct the full causal sequence of events for any trip — useful for debugging, anomaly detection, and lineage visualization.

### Real-Time Path: Stream Processor

The Stream Processor is a stateless Kafka consumer that maintains no on-disk state between runs. It subscribes to all eight topics with a dedicated consumer group, processes events in 100-millisecond windows, and publishes to four Redis pub/sub channels: `driver_updates`, `rider_updates`, `trip_updates`, and `surge_updates`.

GPS ping windowing selects either the latest ping per driver per 100ms window (maximum freshness) or a sampled position (configurable). In most configurations, the latest-position policy is used: for each driver, only the last GPS ping received in the 100ms window is published to Redis. This collapses the potentially high GPS ping rate (up to several pings per simulated second at high speed multipliers) to a stable 10 Hz update rate for WebSocket clients.

Event deduplication uses Redis `SET NX` with a one-hour TTL keyed on `event_id`. If the Redis key already exists, the event is a redelivery (Kafka at-least-once semantics can redeliver on consumer restart) and is dropped silently. This idempotency guarantee means that WebSocket clients never see duplicate state updates even during Stream Processor restarts.

### Real-Time Path: WebSocket Delivery

The simulation API subscribes to the four Redis pub/sub channels and fans out to all connected WebSocket clients. On initial WebSocket connection, clients receive a full state snapshot: all currently active drivers (from Redis hash `drivers:*`), all active trips (from Redis hash `trips:*`), and all active surge zones (from Redis hash `surge:*`). These snapshots have a 30-minute TTL — if a client connects more than 30 minutes after the simulation last updated a driver, that driver's state may be absent, which is by design (stale state is not worth delivering).

After the initial snapshot, clients receive incremental updates pushed by the Redis pub/sub subscriber. The push rate is bounded by the Stream Processor's aggregation window, so GPS updates flow at approximately 10 Hz regardless of simulation speed multiplier. At a 100× speed multiplier, 100 simulated seconds pass per wall-clock second, but GPS updates still arrive 10 times per wall-clock second on the client. The deck.gl animation layer interpolates between positions, making movement appear smooth even at high update rates.

### Batch Path: Bronze Ingestion

Bronze Ingestion is a Kafka consumer that writes raw events to Delta Lake tables on S3/MinIO in micro-batches of approximately 10 seconds. Each Kafka topic maps to a corresponding Bronze Delta table. All eight tables share a common schema:

- `_raw_value` (string): the verbatim UTF-8 JSON payload, exactly as received from Kafka — no parsing, no field extraction
- `_kafka_partition` (integer): Kafka partition the message arrived on
- `_kafka_offset` (long): Kafka offset within the partition
- `_kafka_timestamp` (timestamp): Kafka broker-assigned timestamp
- `_ingestion_timestamp` (timestamp): wall-clock time when Bronze Ingestion wrote the record
- `_ingestion_date` (date): partition key for Hive-style date partitioning

This verbatim storage design preserves the full fidelity of the original event for downstream parsing. No fields are extracted or mutated at Bronze ingestion time. Bronze is an append-only raw landing zone — if a Silver parse macro has a bug, the Bronze data is always available for reprocessing without any information loss.

Malformed messages — those that fail JSON Schema validation or cannot be decoded as UTF-8 — are routed to per-topic DLQ Delta tables (e.g., `bronze_trips_dlq`, `bronze_gps_pings_dlq`). DLQ records include the raw bytes, the validation error message, and the same Kafka provenance fields. The Airflow DLQ monitoring DAG checks these tables every 15 minutes and triggers an alert if error rates exceed configured thresholds.

Delta Lake tables written by Bronze Ingestion are not automatically discoverable by Trino. A registration step calls Trino's `delta.system.register_table` stored procedure to add each table to the metastore. This step is idempotent (re-registering an already-registered table is a no-op) and runs every 10 minutes in production via a Kubernetes CronJob.

### Silver Transformation: DBT Incremental Models

Airflow triggers the Silver DBT run hourly via a dedicated DAG. A `ShortCircuitOperator` checks whether any new Bronze data has arrived since the last run; if not, it short-circuits the DAG to skip the DBT execution, reducing unnecessary processing and avoiding empty incremental runs.

Each Silver staging model (`stg_trips`, `stg_gps_pings`, `stg_driver_status`, `stg_surge_updates`, `stg_ratings`, `stg_payments`, `stg_drivers`, `stg_riders`) reads from the corresponding Bronze table using DBT's incremental materialization. The incremental predicate filters Bronze records with `_ingestion_timestamp > (SELECT MAX(_ingestion_timestamp) FROM this)`, so each run processes only newly arrived records.

Within each staging model, the `_raw_value` JSON string is parsed into typed columns using adapter-dispatched macros. For the DuckDB target, the macro emits DuckDB's `json_extract_string()` and `json_extract()` functions. For the Glue target, the macro emits Spark SQL's `get_json_object()` and `from_json()` functions. The business logic of the model — which fields to extract, how to cast timestamps, which enum values to validate — is identical between targets.

Deduplication uses DBT's `unique_key` configuration with `event_id`, causing DBT to emit `MERGE` (or `INSERT OR REPLACE` for DuckDB) statements that replace existing records with the same `event_id`. This handles Kafka at-least-once redeliveries at the Silver layer even if they were not caught by the Stream Processor's Redis dedup (which uses a different TTL window).

Great Expectations validates Silver tables after each DBT run via an Airflow task. Eight validation suites check:
- `event_id` uniqueness (no duplicates should survive deduplication)
- Enum field ranges (e.g., `status` must be in the valid set of trip states)
- GPS coordinate bounds (latitude within São Paulo's bounding box)
- Timestamp ordering (`event_timestamp` ≤ `published_at` ≤ `_ingestion_timestamp`)
- Field completeness (required fields must be non-null)

Soft failure via the `mostly` parameter allows a configurable percentage of rows to fail without blocking the pipeline. For example, GPS coordinate bounds validation uses `mostly=0.99`, allowing 1% of pings to be outside São Paulo's bounding box (simulated GPS noise near borders) without failing the suite.

### Gold Transformation: Star Schema

The Gold DBT run is triggered by a successful Silver completion (daily in production, but can be run more frequently in development). It performs a full-refresh rebuild of the entire star schema, reading from Silver staging tables as sources.

**Dimension tables:**

`dim_drivers` and `dim_payment_methods` use SCD Type 2 to track attribute changes over time. The implementation uses window functions (`LAG()` and `LEAD()`) over Silver `stg_drivers` and `stg_payments` events, ordered by `event_timestamp` per entity. For each version of a driver record (each `driver_profiles` event in Silver), the model derives `valid_from` (the event's timestamp), `valid_to` (the next event's timestamp, or `9999-12-31` for the current version), and `current_flag` (1 if the record is the most recent, 0 otherwise). This implementation does not require dedicated slowly-changing dimension tooling — it is pure SQL window functions, making it portable across both DuckDB and Glue targets.

`dim_riders` is a simpler dimension that takes the latest version of each rider's attributes without history tracking, since rider attributes change less frequently and the analytics questions of interest (trip counts by rider cohort) are answered by the current record.

`dim_zones` is seeded from a static CSV of São Paulo's 96 districts. This seed table is loaded at DBT initialization time and remains stable across runs. Zone attributes include district name, borough, population density bucket, and pre-computed H3 coverage cells for joining with GPS-based analytics.

`dim_time` is generated as a full calendar table covering the simulation's potential time range at hourly granularity, with pre-computed columns for hour-of-day, day-of-week, month, quarter, and São Paulo business-hours flags. This avoids repeated date arithmetic in fact queries.

**Fact tables:**

`fact_trips` joins completed trip events from `stg_trips` to dimension snapshots valid at the time of trip completion. The distance column (`distance_km`) is computed via the Haversine formula applied to pickup and dropoff coordinates from the trip event — not from OSRM route distance, because route distances are not stored in the event payload. The Haversine approximation understates actual road distance by 15-30% (roads are longer than straight-line distance), but provides a consistent, reproducible metric.

`fact_trips` includes `offer_sequence` as a diagnostic column — the number of driver offers sent before assignment. High values indicate matching difficulty in that zone and time window.

**Aggregate tables:**

`agg_daily_driver_performance` pre-aggregates daily metrics per driver: total trips completed, total online minutes (sum of time in all non-OFFLINE states), total on-trip minutes (sum of time in ON_TRIP and EN_ROUTE_PICKUP states), total earnings, average rating, and acceptance rate. The `online_minutes` column is the sum of all logged-in status durations — including AVAILABLE, REPOSITIONING, EN_ROUTE_PICKUP, and ON_TRIP — not just idle time. This matches industry convention where "online" means "logged into the platform" regardless of passenger status.

`agg_hourly_zone_demand` aggregates trip requests, completions, and cancellations per district per hour, joining to `dim_zones` for geographic context and to `dim_time` for temporal attributes. This table feeds the Grafana "zone demand heatmap" panel via Trino.

`agg_daily_revenue` aggregates platform-level revenue metrics: gross bookings, platform commission, driver earnings, average fare, and trip count by vehicle type.

**Anomaly views:**

`anomalies_gps_outliers` is a SQL view (not a Delta table) that identifies GPS pings more than 50 km from São Paulo's centroid — implausible positions that indicate GPS noise, bad routing, or simulation bugs. Because it is materialized as a view, it is computed on-demand by Trino rather than requiring a Delta table registration step. This is intentional: anomaly detection does not need to be stored; it needs to be queryable.

`anomalies_zombie_drivers` identifies drivers who have had completed trip events in Silver but no corresponding GPS pings in the same time window — "zombie" drivers who appear to complete trips without moving. This anomaly pattern catches bugs in the GPS emission logic or gaps in the Kafka consumer group's coverage.

### DBT Dual-Target Design in Depth

The DuckDB target uses `delta_scan()` — a DuckDB extension function that reads Delta Lake transaction logs and Parquet files directly from S3 or local paths. This eliminates the need for a running Hive Metastore or Spark cluster for local development. The `delta_scan()` call takes an S3 URL or local file path, reads the `_delta_log` directory to determine the current table snapshot, and streams Parquet files through DuckDB's vectorized execution engine. Performance is competitive with Spark for dataset sizes under ~100 GB per table, which is more than sufficient for the simulation's output volume.

The Glue target executes DBT models as Python scripts submitted to AWS Glue Interactive Sessions. Glue Interactive Sessions provide a JVM Spark runtime with the Delta connector configured and AWS Glue Data Catalog as the metastore. The `adapter.dispatch` macros emit Spark-compatible SQL (`get_json_object()`, `from_json()`, Hive SerDe functions) for the Glue target and DuckDB-specific functions for the local target. Adding a new target (e.g., Databricks, BigQuery) would require only adding a new macro dispatch entry — the model SQL remains unchanged.

### Trino Query Layer

Trino provides distributed SQL access to all three medallion layers simultaneously. The Delta connector reads `_delta_log` transaction logs to determine the current table snapshot before scanning Parquet files, ensuring Trino always reads consistent table state even during concurrent Bronze writes. Two catalog configurations are supported: a Hive Metastore-backed catalog (backed by RDS PostgreSQL locally) and an AWS Glue-backed catalog (used in the `production-glue` Kustomize overlay). The same SQL queries work against both, since both expose the Delta tables through the standard Trino Delta connector API.

Trino's FILE authenticator uses a password file rendered from bcrypt hashes stored in Secrets Manager. Two accounts exist: `admin` (full catalog access for operators) and `visitor` (read-only access to the delta catalog). Access control rules are evaluated in order in `rules.json`: the `admin` full-access rule appears first; then a `system` catalog deny applies to all users; then a `delta` read-only grant applies; then a catch-all fallback denies everything else. The ordering is critical — if the `system` catalog deny appeared after the catch-all, visitors would have no access at all.

Grafana's Trino datasource plugin connects over HTTP (not the PostgreSQL wire protocol). The plugin sends SQL queries as JSON over Trino's REST API and receives columnar result sets. All Grafana Trino panels use `rawQuery: true` with the `rawSQL` field (capital SQL — this is a plugin convention that differs from other datasources).

## Design Patterns

### Event-Driven Architecture and Correlation

The simulation is the sole producer of domain events, establishing a single source of truth for all downstream systems. This single-producer design is intentional: no service other than the simulation generates or modifies events. Downstream services only read, transform, or forward events — they never produce new business events of their own.

Every event carries three distributed-tracing correlation fields: a `session_id` identifying the simulation run, a `correlation_id` identifying the primary business entity (typically `trip_id`), and a `causation_id` referencing the event that triggered the current one. The `EventFactory` chains causation by reading the previous event's ID from the trip record and writing the new event's ID back. For a complete trip, the causal chain is: `REQUESTED` → `OFFER_SENT` → `DRIVER_ASSIGNED` → `EN_ROUTE_PICKUP` (first GPS ping) → ... → `AT_PICKUP` → `ON_TRIP` (first passenger GPS ping) → ... → `COMPLETED` → `PAYMENT` → `RATING`. Each event's `causation_id` points to the immediately preceding event in this chain.

Two independent consumer groups process the same events without awareness of each other. Kafka's consumer group model handles the fan-out: the broker delivers each partition's events to one consumer within each group, so the Stream Processor group and the Bronze Ingestion group each receive the full event stream independently. Neither service knows the other exists; adding a third consumer (e.g., a real-time anomaly detector) requires only registering a new consumer group — no changes to the simulation or existing consumers.

### State Machine Enforcement

All stateful entities use enum-based state machines with `VALID_TRANSITIONS` dictionaries. Each dictionary maps a current state to the set of states it may legally transition to. Terminal states map to empty sets. Any transition attempt — whether from application logic or an unexpected external command — passes through the same `transition(target)` method that consults `VALID_TRANSITIONS`. If the target is not in the allowed set, a `StateError` is raised immediately. This design makes invalid state transitions impossible to reach by accident rather than merely unlikely, and eliminates entire classes of bugs where an agent ends up in an inconsistent state.

A Kafka event is emitted on every state transition, making the transition log externally observable. If a bug causes an agent to enter an unexpected state, the Kafka event stream shows exactly which transitions occurred in what order, making root cause analysis straightforward. The Silver staging tables preserve these events, so the full state machine trace for any agent is queryable in SQL after the fact.

The trip lifecycle has 10 states: REQUESTED, OFFER_SENT, OFFER_EXPIRED, OFFER_REJECTED, DRIVER_ASSIGNED, EN_ROUTE_PICKUP, AT_PICKUP, IN_TRANSIT, COMPLETED, and CANCELLED. OFFER_EXPIRED and OFFER_REJECTED are distinct terminal states (no further transitions allowed from CANCELLED or COMPLETED) that carry diagnostic information — an expired offer means the driver did not respond in time, while a rejected offer means the driver actively declined.

### Error Handling and Recovery Intent

A typed exception hierarchy organizes all errors by recovery intent. `TransientError` subclasses (network failures, service unavailability, persistence errors, Kafka `BufferError`) are safe to retry with exponential backoff. `PermanentError` subclasses (validation failures, not-found, configuration errors, type mismatches) bypass retry logic entirely — retrying a validation failure would produce the same result indefinitely. `FatalError` signals immediate process shutdown.

Retry utilities accept only `TransientError` subclasses by default. A `PermanentError` propagates immediately without entering the retry loop, ensuring that application bugs surface as errors rather than being silently swallowed by retry logic. The backoff parameters (base delay, max delay, jitter factor, max attempts) are configurable per call site, allowing different retry aggressiveness for different operations (e.g., OSRM route cache misses retry more aggressively than Kafka publish retries).

The Kafka producer handles `BufferError` (internal producer queue full, indicating Kafka backpressure) by dropping the message with a warning log rather than raising an exception. This prevents a transient Kafka pressure event from crashing the SimPy event loop. The design favors simulation continuity over perfect delivery guarantees, which is appropriate for a simulation context where losing 1-2 GPS pings under heavy load is acceptable, but crashing the event loop is not.

### Immutability and Append-Only Design

The immutability principle pervades the system at multiple levels. Agent DNA is frozen using Pydantic's `frozen=True` configuration, making DNA mutations a compile-time error. Bronze tables store `_raw_value` verbatim — no mutation at ingestion time. Events carry UUID identifiers and are never modified after publication. Delta Lake tables use transaction log semantics where writes are new log entries, not modifications to existing Parquet files.

These properties together make the system's state fully reconstructible from the event log. Given a fresh Delta Lake Bronze layer and the original Kafka event stream, the entire Silver and Gold pipeline can be replayed from scratch to produce identical results. This reconstructibility is a safety property: if a Silver transform has a bug, the Bronze data is untouched and can be reprocessed with the fixed transform.

Cross-thread state sharing in the simulation uses frozen dataclasses for snapshot objects. The SimPy thread creates new `AgentSnapshot` instances (with `@dataclass(frozen=True)`) and atomically replaces the reference in a thread-shared dictionary. The FastAPI thread reads the snapshot reference and accesses its fields without any risk of mutation, because `frozen=True` makes field assignment a `FrozenInstanceError` at runtime. This eliminates the need for read locks around snapshot access.

### Repository Pattern and Data Access

The simulation service uses a generic repository pattern with a `BaseAgentRepository[T]` generic base class. The base class provides standard CRUD methods (`get`, `get_all`, `save`, `delete`, `batch_save`) with SQLAlchemy ORM under the hood. `DriverRepository` and `RiderRepository` inherit from this base and add entity-specific query methods (e.g., `find_available_in_zone`, `find_by_status`).

This pattern shields the domain logic from ORM details. The SimPy agent processes work with domain model instances (frozen dataclasses) and call repository methods when they need to persist or retrieve state. Repositories handle the translation between domain models and ORM entities, ensuring that domain logic never directly touches SQLAlchemy Session objects. Checkpoint writes (persisting simulation state for pause/resume) use a delete-all-then-insert-all strategy wrapped in a savepoint context for atomic rollback on failure.

The SQLite database used for checkpointing is local to the simulation container. It is not a shared database — it exists solely to enable pause/resume across simulation API restarts. On resume, the simulation loads agent state from SQLite and re-creates SimPy processes for each persisted agent. The Redis state (driver/trip current state for WebSocket delivery) is rebuilt by replaying status events from SQLite on startup.

### Observability Design and Signal Correlation

Application services export metrics and traces via OpenTelemetry OTLP/gRPC to the OTel Collector. The collector is the single aggregation point for all observability signals:

- **Metrics pipeline**: OTLP metrics → Prometheus remote write endpoint → Prometheus scrape via pull
- **Logs pipeline**: Docker container log files → OTel Collector filelog receiver → Loki push API
- **Traces pipeline**: OTLP traces → Tempo → Tempo-generated span metrics → Prometheus remote write

Tempo generates two additional Prometheus metric families from incoming traces. `traces_spanmetrics_*` provides RED metrics (rate, error rate, duration percentiles) per service-operation combination. `traces_service_graph_*` provides request rates and duration distributions between connected service pairs. Both are remote-written to Prometheus, enabling correlation between application traces and infrastructure metrics in Grafana dashboards.

Each application service maintains an in-process `PerformanceTracker` — a thread-safe rolling window (60-second deque) of observation tuples. The OTel export layer reads the latest snapshot from this rolling window and converts absolute cumulative counts to incremental deltas before constructing OTLP metric data points. This conversion is necessary because OpenTelemetry's sum metrics are cumulative (monotonically increasing), but the `PerformanceTracker` stores absolute rates for the window — so the OTel exporter must compute the delta between the current and previous export cycle.

GPS pings are sampled at 1% for distributed tracing — approximately 1 in 100 GPS pings generates a trace span. At high simulation speeds, the volume of GPS pings would make 100% trace sampling cost-prohibitive. All other event types (trips, status transitions, offers) are traced at 100%. This sampling ratio keeps trace storage manageable while preserving full observability for high-value events.

### Configuration and Credential Management

Pydantic Settings classes with grouped environment variable prefixes govern service configuration. `SIM_*` variables configure the simulation (speed multiplier range, agent counts, zone weights). `KAFKA_*` variables configure the Kafka connection (broker URL, SASL credentials, Schema Registry URL). `REDIS_*` variables configure Redis (host, port, TTL values). Each Settings class is a Pydantic `BaseSettings` model where field types and validators enforce the configuration schema.

A `model_validator(mode="after")` on each Settings class checks that required credentials are present and non-empty after field population. If a required secret is absent, the validator raises a `ValidationError` with a descriptive message and the service refuses to start. This fail-fast behavior prevents silent misconfigurations where a service starts without valid credentials and then fails on the first request — a common source of hard-to-debug production incidents.

In local development, the `secrets-init` container seeds LocalStack Secrets Manager and writes grouped credential files to a Docker volume mounted by all other services. The Pydantic Settings models read from this volume directory. In production, External Secrets Operator reads from AWS Secrets Manager and creates Kubernetes Secret objects, which are mounted at the same directory paths as the local volume — same Settings code, same paths, different secret backends.

## Infrastructure, Deployment & Security

### Local Development Stack

The full stack runs via Docker Compose with four composable profiles. The profiles follow a dependency ordering that must be respected when starting subsets of the stack:

- **`core` profile**: Provides the real-time simulation runtime — Kafka (KRaft), Schema Registry, Redis, OSRM, the simulation API server, the Stream Processor, and the React control panel frontend. Also includes LocalStack (emulates AWS Secrets Manager, S3, SES, DynamoDB, KMS) and the `secrets-init` bootstrapper that seeds credentials into LocalStack. This profile is the minimum viable stack for testing simulation and real-time features.

- **`data-pipeline` profile**: Adds the medallion lakehouse stack — MinIO (S3-compatible local storage), Bronze Ingestion consumer, Apache Airflow (with embedded scheduler and workers), Hive Metastore (backed by local PostgreSQL), Trino (Delta connector configured), and the DBT and Great Expectations execution environments. Requires `core` to be running first (depends on Kafka and Redis).

- **`monitoring` profile**: Adds the full observability stack — Prometheus (scraping simulation and stream processor metrics), Grafana (with pre-provisioned datasources and dashboards), Loki (log aggregation), Tempo (distributed tracing), OTel Collector (multi-signal routing), and cAdvisor (container resource metrics). Can be started independently of `data-pipeline` (monitoring only the simulation layer is a valid development pattern).

- **`performance` profile**: Adds the PID-based Performance Controller that reads Kafka consumer lag and Redis latency from Prometheus and adjusts the simulation speed multiplier. Used when running extended data accumulation sessions at maximum throughput.

All credentials are managed automatically by the `secrets-init` container. No `.env` file editing is required; developers run `docker compose --profile core up` and the stack bootstraps itself.

### Kubernetes Production Architecture

All production workloads run in the `rideshare-prod` namespace. Ingress uses the Kubernetes Gateway API (`GatewayClass` backed by Envoy Gateway) rather than the older Ingress resource, with `HTTPRoute` objects for path-based routing. Each `HTTPRoute` includes a `URLRewrite` filter that strips the path prefix before forwarding to the backend service (e.g., `/simulation/health` is forwarded to the simulation pod as `/health`). This allows all services to share a single wildcard subdomain with path-based routing rather than requiring separate subdomains.

Two production overlay variants exist via Kustomize:

- **`production-duckdb`**: Deploys Hive Metastore backed by RDS PostgreSQL as the Trino catalog. DBT runs via DuckDB against the Delta files on S3. This overlay is used for cost-optimized deployments where Glue Interactive Session costs are a concern.
- **`production-glue`**: Substitutes AWS Glue Data Catalog for Hive Metastore (eliminating the Metastore pods) and runs DBT via Glue Interactive Sessions. This overlay is appropriate for higher-volume or production-intent deployments.

All workload IAM roles use EKS Pod Identity — not IRSA (IAM Roles for Service Accounts). Pod Identity associations are declared in Terraform and bind `(cluster, namespace, service_account)` tuples to IAM roles. Trust policies target `pods.eks.amazonaws.com` with `sts:AssumeRole` and `sts:TagSession`. No OIDC annotation is required on Kubernetes ServiceAccount objects. Pod Identity is the modern EKS-native approach that avoids the OIDC token volume complexity of IRSA.

The React SPA is not served from Kubernetes. It is a static build deployed to an S3 bucket and served via CloudFront CDN with Origin Access Control restricting direct S3 access to CloudFront only. HTML files use `Cache-Control: no-cache, must-revalidate` to ensure clients always get the latest HTML (which includes versioned asset URLs). JavaScript and CSS assets use `Cache-Control: max-age=31536000` (one year) since they are content-addressed by hash. When the platform layer is torn down, the CDN continues serving the SPA, which displays a graceful "platform offline" state.

### Three-Layer Terraform Model

Infrastructure is organized into three Terraform layers with strict dependency ordering:

**Bootstrap layer**: Manages only the Terraform remote state backend — the S3 bucket for state storage and the DynamoDB table for state locking. Run once per AWS account. Never destroyed. This layer exists because Terraform's S3 backend configuration (which references the state bucket) cannot be managed by Terraform itself — bootstrapping the state backend requires local state.

**Foundation layer**: Persistent, cost-free-at-rest resources (~$7.50/month): VPC (with public and private subnets across three AZs), S3 buckets (bronze, silver, gold, checkpoints, frontend SPA, logs, telemetry), ECR container registries (one per service image), IAM roles (workload roles for each service, CI/CD OIDC role for GitHub Actions), Route 53 hosted zone, ACM TLS certificate (wildcard, auto-validated via DNS), CloudFront distribution (SPA delivery with OAC), AWS Secrets Manager secrets (service credentials), Glue Data Catalog databases, DynamoDB table (visitor records), KMS CMK (visitor password encryption), and SES configuration (welcome emails). All compute-free resources — just configuration and metadata.

**Platform layer**: Ephemeral compute (~$0.31/hour while running): EKS cluster with one managed node group (t3.xlarge, running all pods), RDS PostgreSQL (db.t3.micro, Multi-AZ disabled for cost), Application Load Balancer (created by the AWS Load Balancer Controller addon), all Kubernetes workloads via ArgoCD, and EKS addon configurations (VPC CNI, CoreDNS, kube-proxy, Pod Identity agent, Load Balancer Controller). The platform layer depends on the foundation layer for VPC IDs, ECR URLs, IAM role ARNs, and secret names.

### CI/CD Pipeline

Eight GitHub Actions workflows manage the full deployment lifecycle:

1. **CI (continuous integration)**: Runs on every push to `main`. Python linting (ruff), type checking (mypy), unit tests (pytest), frontend lint (eslint), frontend tests (vitest), frontend build (Vite), and API contract validation (OpenAPI spec export + TypeScript type comparison). This workflow gates merges to `main`.

2. **Build images**: Runs on pushes to `main` with path filtering. Detects which service directories changed and rebuilds only the corresponding Docker images. Pushes to ECR with both git SHA tag and `latest` tag. GitHub Actions authenticates to ECR via OIDC web identity federation — no long-lived AWS credentials stored.

3. **Deploy platform**: Full platform provisioning. Runs `terraform apply` on the platform layer. Force-pushes resolved Kubernetes manifests to the `deploy` branch (substituting placeholders: `${AWS_ACCOUNT_ID}`, `${RDS_ENDPOINT}`, `${ACM_CERT_ARN}`, latest ECR image tags). Installs ArgoCD. Waits for phased service convergence across five phases (infrastructure addons → schema registry and Kafka → application services → data pipeline → UI) with per-phase health checks. Reports per-service readiness back to the Lambda function via webhook.

4. **Teardown platform**: Runs `terraform destroy` on the platform layer. Suspends ArgoCD auto-sync first (to prevent ArgoCD from fighting Terraform during resource deletion). Runs with `--auto-approve`. Does not touch the foundation layer.

5. **Soft reset**: Wipes runtime state (Kafka PVCs, MinIO/S3 lakehouse contents, RDS database, Redis cache, monitoring volumes) without destroying the EKS cluster. Temporarily suspends ArgoCD auto-sync during the wipe to prevent self-healing from re-creating resources. Used when a demo session needs a clean slate without the cost of a full teardown and re-deploy cycle.

6. **Build SPA**: Builds the React frontend and deploys to S3, then invalidates the CloudFront distribution cache. Triggered by changes to the `services/control-panel/` directory.

7. **Validate schemas**: Exports the simulation API's OpenAPI spec and compares the generated TypeScript types against the committed `api.generated.ts`. Fails if they differ, enforcing API contract integrity.

8. **Security scan**: Runs `trivy` container scanning on all ECR images and `bandit` on Python source code. Results uploaded as GitHub Actions security advisories.

### The `deploy` Branch as Materialized Artifact

A key architectural pattern in the CI/CD pipeline is the `deploy` branch as a materialized artifact rather than a source branch. The `main` branch always contains placeholder tokens in Kubernetes manifests (e.g., `${IMAGE_TAG}`, `${RDS_ENDPOINT}`, `${AWS_ACCOUNT_ID}`). The CI deploy workflow resolves these placeholders at deploy time (reading values from Terraform outputs and ECR) and force-pushes the fully-resolved manifests to the `deploy` branch with a `[skip ci]` commit message.

ArgoCD watches only the `deploy` branch with `selfHeal: true` and `syncPolicy: automated`. If any Kubernetes resource deviates from the `deploy` branch manifest (e.g., a developer manually scaled a deployment via `kubectl`), ArgoCD detects the drift within three minutes and reverts it. This pattern ensures that the cluster state is always exactly what CI produced — manual drift cannot persist.

The `deploy` branch is force-pushed on every deploy, replacing the previous state entirely. This means `deploy` branch history is not meaningful; it is always a single commit containing the current resolved manifest set. The `main` branch retains the full history and the source of truth for template tokens.

### On-Demand Session Lifecycle

The auth-deploy Lambda function (Python 3.13, deployed as an AWS Lambda Function URL with IAM authentication from CloudFront) manages the demo session lifecycle. It handles four action types:

- **`deploy`**: Validates the caller's API key, checks that no session is already active in DynamoDB, dispatches the GitHub Actions deploy workflow via REST API, creates a session record in DynamoDB, and schedules an auto-teardown EventBridge rule (maximum two-hour session by default). Returns the session ID and estimated readiness time.

- **`teardown`**: Validates the caller's API key, dispatches the teardown GitHub Actions workflow, deletes the EventBridge auto-teardown rule, and marks the DynamoDB session as TEARDOWN_IN_PROGRESS. Returns immediately; teardown completes asynchronously.

- **`extend`**: Validates the caller's API key, extends the EventBridge auto-teardown rule by 15 minutes (up to a maximum total session duration of four hours), and updates the DynamoDB session record.

- **`provision-visitor`**: Validates the email format and password strength, checks DynamoDB for an existing record with the same email hash, creates a DynamoDB item with the KMS-encrypted password ciphertext, derives a bcrypt hash for Trino, stores it in Secrets Manager, and sends a SES welcome email. Returns HTTP 201 on success or HTTP 207 (Multi-Status) if the DynamoDB write succeeded but SES failed — the visitor record is durable even if the email bounces.

The DynamoDB `deploy_progress` field tracks per-service readiness as a JSON object. The CI deploy workflow calls the Lambda's webhook endpoint after each service comes healthy, updating `deploy_progress`. The control panel polls the Lambda status endpoint and displays a live readiness grid (15 services, each with a traffic-light indicator) for the operator to monitor.

### Two-Phase Visitor Provisioning

Visitor self-registration is decoupled from platform availability via a two-phase provisioning design that tolerates complete platform unavailability during registration.

**Phase 1 (`provision-visitor`)** runs without any platform infrastructure:
1. Validates email format and password strength (minimum 12 characters, character class requirements)
2. Checks DynamoDB for an existing record keyed by `sha256(email.lower())` — prevents duplicate registrations
3. Writes a DynamoDB item: email hash as partition key, registration timestamp, `kms_encrypted_password` (the plaintext password encrypted by KMS CMK), and `status=PROVISIONED`
4. Derives a bcrypt hash (cost factor 10) of the password and writes it to Secrets Manager (for Trino FILE authenticator)
5. Sends a SES welcome email with the visitor's login credentials for the platform services
6. Returns HTTP 201 (or 207 if SES failed after DynamoDB write succeeded)

**Phase 2 (`reprovision-visitors`)** is called by the deploy workflow after all 15 services report healthy:
1. Scans the full DynamoDB visitor table
2. For each visitor record, decrypts `kms_encrypted_password` via KMS
3. Creates service accounts in parallel:
   - Grafana: Admin API `POST /api/admin/users` with `Admin` role
   - Airflow: REST API `POST /auth/users` with `Viewer` role
   - MinIO: Admin SDK `add_user()` + `set_policy()` with read-only bucket policy
   - Simulation API: `POST /auth/register` with `viewer` role
4. Updates the DynamoDB record's `status` to `ACTIVE`

If the platform is torn down and redeployed, Phase 2 runs again and re-creates all visitor accounts from the durable DynamoDB records. Visitors do not need to re-register between sessions. This idempotent re-provisioning is a key reliability property of the system.

### Authentication Architecture

The simulation API supports two authentication mechanisms. REST requests pass an API key via the `X-API-Key` header. WebSocket connections pass the key via the `Sec-WebSocket-Protocol: apikey.<key>` subprotocol header, because browsers cannot send custom headers during the WebSocket upgrade handshake — the subprotocol field is the only standardized channel for passing metadata during upgrade.

Two key types are accepted by the API key validation dependency (`Depends(verify_api_key)`):

- **Admin key**: A static key sourced from Secrets Manager at startup. Always resolves to `AuthContext(role="admin", email="admin")`. Grants full mutation access to all endpoints.
- **Session keys**: Keys with a `sess_` prefix. Validated by a Redis hash lookup against a session record with a 24-hour TTL. The session record stores `role` and `email`, created by `POST /auth/login`. Session keys support `viewer` role for provisioned visitors.

The Trino query engine mirrors this dual-account design at the data layer. Grafana connects to Trino as the `visitor` account, which has read-only access to the `delta` catalog. Direct Trino CLI access for operators uses the `admin` account. This prevents Grafana dashboard queries from accidentally running DDL or deleting data.

### Authorization

Two roles exist — `admin` and `viewer` — enforced by FastAPI dependency injection. All mutation endpoints use `Depends(require_admin)`, which reads the `AuthContext` from the request scope and raises HTTP 403 if `role != "admin"`. The following operations require admin: starting/pausing/resuming/stopping the simulation, setting the speed multiplier, spawning driver or rider agents, puppet (direct agent control) commands, and user provisioning. Read-only endpoints — current simulation state, agent lists, trip lists, metrics — accept any valid key including viewer keys.

The React frontend reads the user's role from session storage and conditionally renders admin-only controls (Start/Stop/Pause buttons, agent spawn sliders, speed multiplier input) only when `role === "admin"`. This is a UI convenience, not a security control — the server-side `require_admin` dependency is the authoritative enforcement point.

### Rate Limiting

The simulation API applies per-key rate limiting using `slowapi` (a FastAPI wrapper around `limits`). Per-endpoint limits are configured by sensitivity:

- **`/auth/login`**: 10 requests per minute per IP address. Lower limit to mitigate password brute-force attacks.
- **`/auth/register`**: 20 requests per minute per IP address. Higher than login since it is called programmatically by the provisioning Lambda.
- **WebSocket connections**: 5 new connections per 60-second sliding window per API key, enforced via a Redis sorted-set counter. Connections that exceed the limit receive closure code 1008 (Policy Violation).
- **All other endpoints**: 100 requests per minute per API key (falling back to IP address for unauthenticated or invalid-key requests).

Rate limit counters are stored in Redis with appropriate TTLs. The `slowapi` library uses a module-level rate limiter singleton, which requires explicit test isolation: unit tests that exercise rate-limited endpoints purge the module cache entry for the rate limiter between tests to prevent counter state from leaking between test cases.

### Security Headers and TLS

Every HTTP response from the simulation API carries security headers applied by a dedicated `SecurityHeadersMiddleware` class that runs on every request before the route handler:

- `Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'`
- `Strict-Transport-Security: max-age=31536000; includeSubDomains` (HSTS, one-year max-age)
- `X-Frame-Options: DENY` (prevent clickjacking via iframe embedding)
- `X-Content-Type-Options: nosniff` (prevent MIME sniffing)
- `Referrer-Policy: strict-origin-when-cross-origin`
- `Permissions-Policy: geolocation=(), microphone=(), camera=()`

In production, CloudFront enforces TLS 1.2 minimum with an ACM wildcard certificate (`*.ridesharing.portfolio.andresbrocco.com`). The SPA is served with Origin Access Control that restricts S3 bucket policy to CloudFront only — direct S3 URL access returns HTTP 403.

### PII Masking

All log records emitted by the simulation service pass through a `PIIFilter` that applies regex substitution to mask email addresses (matched by RFC 5322-style regex, replaced with `[EMAIL]`) and phone numbers (matched by Brazilian phone format regex, replaced with `[PHONE]`) before records reach any handler. This filter operates on plain string messages and formatted exception strings; structured JSON log fields are not scanned.

Profile events (`driver_profiles`, `rider_profiles`) carry synthetic PII generated by Faker with Brazil locale: full names, email addresses, CPF numbers (Brazilian individual taxpayer registration), and phone numbers. These are realistic-looking but entirely synthetic. The PII masking ensures that even synthetic PII does not appear in log streams shipped to Loki, where it could be accidentally treated as real PII by downstream monitoring tools or data governance systems.

### Secrets Management Architecture

No credentials are hardcoded anywhere in the codebase. In local development, LocalStack emulates AWS Secrets Manager and S3. The `secrets-init` container seeds LocalStack with all required secrets (Kafka SASL credentials, Redis password, simulation API key, database passwords, Trino bcrypt hashes) and writes grouped credential files to a shared Docker volume mounted at `/run/secrets/` in all other service containers. Pydantic Settings models read from this volume.

In production, External Secrets Operator (ESO) reads from AWS Secrets Manager and materializes Kubernetes Secret objects. The ESO `SecretStore` configuration points to the AWS Secrets Manager endpoint (different from LocalStack's endpoint in local development, but using the same IAM-based access pattern). Kubernetes Secret objects are mounted at `/run/secrets/` in pods — the same path as the Docker volume in local development. This path consistency means the Pydantic Settings code is identical between environments.

Visitor passwords follow a dual-hash design: a bcrypt hash (cost factor 10, salt auto-generated by `bcrypt.gensalt()`) for the Trino FILE authenticator, and KMS envelope encryption for the plaintext password stored in DynamoDB. KMS envelope encryption uses the CMK to encrypt a data key, which encrypts the plaintext — this is the standard AWS KMS envelope encryption pattern. Decryption requires the CMK permission (`kms:Decrypt`), which only the Lambda function's IAM role has. The Kubernetes workloads have `kms:GenerateDataKey` permission for encrypting new data but not `kms:Decrypt` for reading existing visitor passwords.

### Audit Logging

Every HTTP request to the simulation API generates a structured access log entry at `INFO` level containing: method, path, status code, response time (milliseconds), user identity, and role. For session keys, identity resolution performs a Redis hash lookup to retrieve the email stored in the session record. For the admin key, identity resolves to the string `"admin"`. All logs are structured JSON and are shipped to Loki via the OTel Collector's filelog receiver, which tails Docker container log files and parses JSON fields for Loki label extraction.

An operator-only Grafana dashboard aggregates per-service visitor activity across all platforms. It queries: Airflow login history (via direct Airflow metadata PostgreSQL queries through the Airflow datasource), Grafana user session events (via Loki queries against Grafana's audit log), simulation API access logs (Loki queries filtered by the `simulation` service label), Trino completed query events (Loki queries parsing Trino's query completion log lines), and MinIO audit events (Loki queries against MinIO's JSON audit log). This dashboard is provisioned into a dedicated `Admin` Grafana folder visible only to accounts with the `Admin` role, not to visitor-role accounts.

### Cost Model

The on-demand architecture produces a predictable cost model:

- **Foundation-only (no active platform)**: ~$7.50/month. Dominated by RDS storage snapshots retained for 7 days, S3 standard storage for Delta Lake data, Route 53 hosted zone ($0.50/month), and ACM certificate (free but the CloudFront distribution costs ~$1/month minimum).
- **Platform running**: ~$0.31/hour. Dominated by the t3.xlarge managed node group ($0.19/hr) and RDS db.t3.micro ($0.02/hr), with the remainder in ALB, NAT gateway, and data transfer.
- **Typical portfolio usage (one 4-hour demo per week)**: ~$5/month platform + $8/month foundation = ~$13/month total.
- **Worst-case (platform running continuously)**: ~$230/month. The session time-boxing (2-hour maximum, auto-teardown via EventBridge) prevents this by construction.

The platform uses a single managed node group (t3.xlarge) with all pods scheduled on one node. This over-provisions per-pod resources compared to a multi-node setup but dramatically reduces the node overhead cost: three t3.medium nodes ($0.16/hr × 3 = $0.48/hr) cost more than one t3.xlarge ($0.19/hr) while providing less total memory (12 GB vs. 16 GB). Single-node scheduling simplifies pod affinity requirements and eliminates cross-AZ data transfer costs between nodes.

## Testing

### Testing Philosophy

The testing strategy reflects the platform's layered architecture: each layer is tested in isolation first, then in composition. The test pyramid has a broad base of fast unit tests (no external dependencies), a middle tier of integration tests (live Docker Compose stacks, slower but higher fidelity), and a narrow peak of data quality tests (DBT and Great Expectations, run against actual transformed data). Performance tests sit outside the pyramid — they are measurement exercises rather than pass/fail correctness tests.

The simulation service receives the deepest test coverage because it is the system's most complex component: the sole event producer, the only service with substantial domain logic (state machines, DNA-based decisions, geospatial algorithms, SimPy process coordination), and the component whose bugs would propagate to every downstream layer.

### Simulation Unit Tests

The simulation test suite is organized into approximately 15 subdirectories matching the service's module structure: `test_agents/` (driver and rider behavior), `test_engine/` (SimPy process model, ThreadCoordinator), `test_state_machines/` (transition enforcement, terminal state behavior), `test_matching/` (H3 spatial lookup, composite scoring), `test_geospatial/` (Haversine, H3 cell conversion, OSRM caching), `test_producers/` (Kafka, Redis publish paths), `test_persistence/` (SQLite checkpoint, repository CRUD), `test_surge/` (pricing calculator), `test_dna/` (Faker generation, frozen model semantics), and `test_api/` (FastAPI endpoint behavior).

The test suite uses several isolation techniques to manage the service's module structure. Engine imports are patched via `sys.modules` to prevent the heavyweight SimPy engine from initializing in API tests — since the engine creates database connections and spawns threads at import time, these must be stubbed before the test module imports the API app. Rate limiter state is isolated between tests by purging the `slowapi` module cache entry between each test that exercises rate-limited endpoints, since the limiter stores counters on a module-level singleton. FastAPI dependency overrides inject fixed `AuthContext` values via `app.dependency_overrides[verify_api_key]` without touching Redis.

The DNA system uses seeded Faker instances for deterministic agent generation in tests: `Faker("pt_BR", seed=42)` produces identical DNA values across all environments. This means test assertions on DNA field values are stable — a test that asserts a specific driver's acceptance rate will produce the same value in local development, CI, and macOS ARM architecture.

### Lambda Unit Tests

Lambda unit tests (approximately 15 test files) cover the visitor provisioning paths including: successful full provisioning (DynamoDB write + Secrets Manager bcrypt write + SES send), HTTP 207 partial success (DynamoDB and Secrets Manager succeed, SES fails), duplicate registration detection (same email hash in DynamoDB), deploy action (dispatching GitHub Actions workflow, DynamoDB session creation, EventBridge scheduler creation), teardown action, extend action (EventBridge rule update, DynamoDB timestamp update), and deploy progress updates. All tests use `moto` for AWS service mocking — no live AWS credentials or real infrastructure required.

### Frontend Unit Tests

Approximately 30 TypeScript test files cover React components and hooks: the map component (deck.gl layer rendering, marker positioning), the control panel (admin control visibility by role, simulation controls, agent spawn sliders), the landing page (visitor access form, provisioning form), inspector popups (driver and trip detail drawers), the architecture diagram component, custom hooks (`useWebSocketState`, `useAgentState`, `useSimulationLayers`, `useApiHealth`, `useDraggable`), the Lambda service client (deploy/teardown triggering, visitor provisioning with HTTP 207 handling, credential-hiding behavior), and utility functions (Haversine distance, H3 formatting, time formatting).

Vitest with Testing Library provides the test runner and component rendering utilities. `jsdom` simulates the browser environment. External dependencies with native code or complex browser APIs are mocked via `vi.mock()`: MapLibre GL (the map renderer), deck.gl (the WebGL layer system), and the WebSocket global. These mocks allow component tests to run without a real browser or GPU context.

### API Contract Testing

A dedicated test validates the API contract boundary between the Python FastAPI server and the TypeScript frontend. It exports the current OpenAPI spec from the FastAPI application and compares the generated TypeScript types (via `openapi-typescript`) against the committed `api.generated.ts` file in the frontend service. If they differ — indicating that the API spec changed without regenerating the TypeScript types — the test fails. This test runs in CI on every push to `main` and prevents the drift where the Python API evolves (new fields, changed types, new endpoints) but the TypeScript client is not updated to match, which would cause compile-time type errors in the frontend.

### Integration Tests

Seven integration test files verify the full-stack data pipeline against live Docker Compose services. Tests are organized by verification scope:

- **Feature journey tests**: End-to-end trip lifecycle from simulation API through Kafka consumer groups to Bronze ingestion to Silver transformation to Gold aggregation to Trino query. A single test spawns agents, waits for trips to complete, triggers the DBT pipeline, and asserts that Gold `fact_trips` contains the expected rows with correct field values.

- **Data flow correctness**: Verifies that event fields are preserved correctly through each transformation. Picks a random `event_id` from Kafka, finds it in Bronze `_raw_value`, confirms the Silver staging table parsed it correctly, and confirms any Gold aggregation reflects it.

- **Cross-phase consistency**: Verifies that Bronze, Silver, and Gold agree quantitatively. Trip counts in Gold `fact_trips` must be ≤ Silver `stg_trips` (some trips may not yet be in Gold due to the daily run schedule) must be ≤ Bronze (all trips must exist at Bronze).

- **External integration**: Verifies Prometheus metric cardinality (at least the expected number of labeled metrics present), Grafana datasource connectivity (all five datasources return healthy status via the Grafana health API), and Loki label index (expected service labels are indexed).

- **Resilience**: Tests DLQ routing (injecting a malformed event via the configurable malformed rate, confirming it appears in the DLQ table and not in the main Bronze table), and consumer lag recovery (pausing Bronze Ingestion, letting Kafka lag accumulate, restarting, verifying lag drains to zero within a timeout).

- **Data quality regression**: Asserts known invariants: GPS pings must be within São Paulo's bounding box (99th percentile), surge multipliers must be in [1.0, 5.0], completed trip counts must be ≤ total trip request counts, accepted-offer counts must equal completed-trip counts (within DLQ tolerance).

Integration tests declare Docker Compose profile dependencies via a custom `@pytest.mark.requires_profiles(["core", "data-pipeline"])` marker. The session-scoped fixture reads these markers from the collected test items, starts only the required profiles, waits for health checks, runs tests, and tears down. This means running `pytest tests/integration/test_surge.py` starts only the `core` profile (surge pricing is real-time and doesn't require the data pipeline), while `pytest tests/integration/test_medallion.py` starts both `core` and `data-pipeline`.

Controlled event generators in the integration fixtures produce structured test events with non-colliding IDs (using `uuid4()` with a test-session prefix) for each test function. This allows parallel test execution via `pytest-xdist` without state collision in Kafka topics or Delta tables.

### DBT Data Quality Tests

Thirteen SQL test files and one pytest integration file enforce data quality contracts on the Gold layer. These run via `dbt test` and are separate from the Great Expectations suites (which run via Airflow after each DBT run).

Generic tests check structural correctness:
- SCD Type 2 validity: non-overlapping `valid_from`/`valid_to` ranges per entity, exactly one record with `current_flag=1` per entity
- Surrogate key uniqueness across all dimension and fact tables
- Referential integrity: `fact_trips.driver_sk` must exist in `dim_drivers.driver_sk`

Singular tests assert business logic:
- Fare calculation correctness: `base_fare * distance_factor * surge_multiplier` must equal `total_fare` within floating-point tolerance
- Surge multiplier bounds: all surge values must be in [1.0, 5.0]
- GPS outlier detection: the `anomalies_gps_outliers` view must match what a direct Haversine distance calculation identifies as outliers
- Zombie driver detection: `anomalies_zombie_drivers` must identify drivers whose trip count in `fact_trips` exceeds their GPS ping count in Silver
- Dimension completeness: all 96 São Paulo districts must be present in `dim_zones`
- Revenue consistency: sum of `fact_trips.total_fare` must equal sum of `stg_payments.amount` within a configurable tolerance (some difference is expected for cancelled-trip payments)

### Great Expectations Validation

Eight Silver validation suites cover all staging tables. Suites are registered in the Great Expectations context and executed by Airflow tasks after each Silver DBT run.

Key expectations per suite:

- `stg_trips`: `event_id` uniqueness (zero duplicates after deduplication), `status` is in the valid enum set, `pickup_lat`/`pickup_lng` within São Paulo bounding box, `offer_sequence` is a positive integer.
- `stg_gps_pings`: `driver_id` is non-null, `latitude`/`longitude` within São Paulo bounding box (`mostly=0.99` — 1% outlier tolerance), `accuracy_meters` is between 1 and 50.
- `stg_driver_status`: `status` is in the valid driver state enum, `driver_id` is non-null, no future `event_timestamp` values (within a 5-second clock skew tolerance).
- `stg_surge_updates`: `surge_multiplier` is between 1.0 and 5.0, `h3_zone` is a valid H3 cell index at resolution 7.
- `stg_ratings`: `rating` is between 1 and 5 (integer), `rater_type` is in `["rider", "driver"]`.
- `stg_payments`: `amount` is positive, `currency` is `"BRL"`, `payment_method` is in the valid enum set.
- `stg_drivers` and `stg_riders`: Required PII fields (`email`, `phone`) are always present but masked in logs; `dna_*` fields for key behavioral parameters are within valid ranges.

The `mostly` parameter on coordinate bound expectations acknowledges that GPS noise near district boundaries will occasionally produce coordinates technically outside São Paulo's administrative bounding box but physically nearby. Setting `mostly=0.99` rather than `mostly=1.0` prevents spurious pipeline failures from GPS noise while still catching systematic outliers (e.g., all pings from a zone being placed in the Atlantic Ocean, which would indicate a bug in the coordinate transformation).

### Performance Testing

A custom performance measurement framework runs four scenario types outside the standard pytest suite:

- **Baseline scenario**: Measures steady-state container resource usage (CPU, memory, network I/O) at nominal simulation speed (1× multiplier) with 50 active drivers and 30 riders. Establishes the resource floor for capacity planning.

- **Stress scenario**: Ramps to maximum agent count (500 drivers, 200 riders) and measures resource usage, Kafka consumer lag, and event throughput. Identifies the agent count ceiling before Kafka lag becomes unsustainable.

- **Speed scaling scenario**: Runs the simulation at 1×, 10×, 50×, and 100× speed multipliers with a fixed agent count (100 drivers, 50 riders) and measures event throughput at each multiplier. The throughput curve is fitted to the Universal Scalability Law: `C(N) = N / (1 + σ(N-1) + κN(N-1))`, where `N` is the speed multiplier, `σ` is the serialization/contention coefficient, and `κ` is the coherency coefficient. High `κ` (>0.001) indicates cross-thread coordination overhead; high `σ` (>0.1) indicates serialization bottlenecks in the SimPy event loop or Kafka produce path.

- **Duration leak scenario**: Runs the simulation for an extended period (4 hours wall-clock at 10× multiplier) and samples container memory every 60 seconds. Fits a linear regression to the memory trace. A slope >1 MB/minute indicates a memory leak in the simulation process (e.g., unbounded DNA cache growth, agent snapshot accumulation). This test was added after discovering that the OSRM route LRU cache was not evicting expired entries correctly, causing memory growth proportional to the number of unique O-D pairs over the session lifetime.

Results are exported as Plotly HTML charts (throughput vs. multiplier, resource usage over time, USL fit visualization) and a structured JSON report. The analysis layer generates markdown summaries suitable for inclusion in portfolio documentation.
